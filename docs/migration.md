# Migration guide for open-condo-software condo platform

Our platform is growing and requires more performance and horizontal scalability. 

This document will outline the steps required to migrate to major versions.

- [# From version v2.x.x to version v3.x.x](#from-version-v2xx-to-version-v3xx)


## From version v2.x.x to version v3.x.x

We are going to migrate key-value storage from single node setup (at the moment it is the Redis engine) to clustered ValKey. The reasons for this decision are the following facts:

- Easy horizontal storage scaling
- High fault tolerance
- Possibility of placing data in several data centers/different machines

In this case, the first thing that is incompatible with the Redis/Valkey cluster is virtual databases. 

So if you have plans to move your key-value storage from standalone instance to a clustered one is add prefixes to all of your applications to make sure you don't have any correlations or corruptions of the data.

For that reason all keys generated by Redis/Valkey adapter will be prefixed with application name. So, if you call ```redis.set('test', 1)``` result will be ```condo:test -> 1``` 

Enough of the lyrics, let's get down to action!

### Application running in production

1) Add ```REDIS_FALLBACK_ENABLED=true``` to your application environment via .env file or something similar
2) Restart app -> now app automatically prefix all write operations, but read operations will be retried without prefix if that key wasn't found.
3) Run migration, that contains script ```rename-redis-keys.js``` via command ```yarn workspace @app/condo migrate``` (for condo application it'll be  ```20250124162345-0446_auto_20250124_1123.js```)
4) After migration succeeded -> remove ```REDIS_FALLBACK_ENABLED=true``` env and restart application again 

### Local development / low traffic application

If you don't need to perform updates without downtime, you can simply perform migrations as usual.

```yarn workspace @app/condo migrate```
